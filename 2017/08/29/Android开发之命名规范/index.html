Android开发之命名规范

1.项目规范

1.1 项目结构

新的项目构建需要遵循Gradle项目结构，可以在官方Gradle插件用户向导了解详细信息. 通过这个项目学习更多用法。

1.2 命名方式

1.2.1 类的命名

类的命名应按照驼峰式结构,如：UpperCamelCase.

对于继承自Android系统组件的类，需要以组件名称作为文件名结尾,例如：SignInActivity, RegisterFragment, ImageUploaderService, ChangePasswordDialog。

1.2.2 资源文件

资源文件应当以小写和下划线的方式来命名。

1.2.1.2 图片资源

1.2.2.2 布局文件

布局文件名称应该与其对应的Activity或Fragment的名称一致，但需要把组件名称从类名的后缀变成布局文件的前缀。比如SignInActivity对应的布局文件名称应当是:activity_sign_in.xml。

Component Class Name Layout Name
Activity UserProfileActivity activity_user_profile.xml
Fragment SignUpFragment fragment_sign_up.xml
Dialog ChangePasswordDialog dialog_change_password.xml
AdapterView item — item_person.xml
Partial layout — partial_stats_bar.xml

需要注意的一个情况是，如果是通过Adatpterinflate的布局文件，需要以item_开头。

另外一个需要注意的是，当一个布局将作为另一个布局的一部分时，需要以partial_开头。

1.2.2.3 菜单

与资源文件类似，菜单的命名同样应该匹配组件名称，例如我们定义一个用在UserActivity的菜单，则应当把其命名为activity_user。

因为他们本身已经被放置在menu目录中，所以菜单文件名中不需要包含menu关键字。

1.2.2.4 Values

values目录下的资源文件名称应当为复数形式，例如：strings.xml, styles.xml, colors.xml, dimens.xml, attrs.xlm

1.3 XML资源命名

1.3.1 ViewId的命名

布局文件中，View Id使用小写字母m开头, 例如：mConfirmBtn, mSubmitBtn, mCancelView。

很多规范建议使用控件类型_开头如text_sign_in，也有不少人建议loginButton这种命名方式。但是这种方式：

  • 引入到代码中需要经过另一层对应关系才能找到
  • 下划线过多的话会对代码阅读产生干扰
  • 某些第三方库如AndroidAnnotation会直接通过布局中的viewId按照同名查找自动赋值给类变量，增加了手动维护的成本。

1.3.2 Strings 字符串命名

  1. 一般情况下Strings 命名需要以其所在模块的业务含义单词作为前缀，例如：registration_mobile_hint,login_password_required_hint。

    另外每个模块需要连续集中放置，不可随意在文件尾部随意追加。

  2. 如果字符串不属于任何一个业务模块，则需要将其归类，按照以下约定命名：
Prefix Description
error_ An error message
msg_ A regular information message
title_ A title, i.e. a dialog title
action_ An action such as “Save” or “Create”

1.3.2 主题和样式

以大写字母开头的驼峰式（UpperCamelCase）来命名：ClickableImageLayoutStyle

1.4.1 方法命名和规范

1.4.1.1 方法名和返回值

方法名称是以小写字母开头的驼峰式结构。定义方法时需要有明确的作用域，根据名称即可知其作用。方法的名称需要与其返回值对应，例如public void createFruit()显然是不合理的。create是创建的意思，那么就应该具有返回值而非void，而且创建出来的结果也应当是Fruit而不是Apple，最后作用域是否应当为public也需要按照对象的实际场景来确定。相反地，private Fruit onDestroy()的返回值也没有任何意义。

1.4.1.2 方法参数

功能性的方法应当尽可能通过参数传入完成单一功能。尽可能少地引用类的成员变量，达到方法内聚的目的。

对于未引用成员变量的方法，应当优先考虑声明为static。
